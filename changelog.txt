v5.2.3 — Security Audit & Critical Bug Fix Round (PATCH)
----------------------------------------------------------------------------------
### Root Cause Analysis (RCA):
- **[CRITICAL] `existingTx` ReferenceError:** `payments.ts` line 262 referenced `existingTx` instead of `previousStatus` — caused silent crash in manual transaction verify path.
- **[CRITICAL] Points double-spend:** `payments.ts` non-atomic points deduction — concurrent requests could spend same points twice.
- **[CRITICAL] `updateSellerBalances` non-idempotent:** Called multiple times (iPaymu callback + auto-cleanup) could duplicate seller balance credits.
- **[HIGH] TOCTOU race in stock restore:** `stock.js` SELECT-then-UPDATE pattern allowed concurrent restores to phantom stock.
- **[HIGH] `total_amount` permanently mutated by partial points:** Partial points payment reduced `total_amount` in DB — skewed financial reports forever.
- **[MEDIUM] Missing auth on critical endpoints:** `/api/digital/deposit` (open to all), `/api/digital/order` (open), `/api/transactions/cancel` (no ownership check).
- **[MEDIUM] Quantity validation bypass:** Cart store allowed negative/NaN quantities via URL manipulation.
- **[LOW] Misleading `getUserId` function name:** Actually returns auth token, not user ID — confusion risk.
- **[LOW] API key prefix leaked in error logs:** Digiflazz `bc` prefix exposed in console.error.

### Perbaikan:
- **[FIX] `src/routes/payments.ts` (existingTx):** Hapus referensi `existingTx`, pakai `previousStatus` dari destructured `txRecord`.
- **[FIX] `src/routes/payments.ts` (points double-spend):** Atomic deduction via `.gte("loyalty_points", amount)` + `.select()` validation — rollback if saldo changed.
- **[FIX] `src/services/payment.js`:** Idempotency guard with `metadata->>balances_updated` flag — skip if already credited.
- **[FIX] `src/services/stock.js`:** TOCTOU fix on `deductTransactionStock` — atomic conditional update `.filter('metadata->>stock_restored', 'eq', 'true')` before restore.
- **[FIX] `src/services/stock.js`:** TOCTOU fix on `restoreTransactionStock` — atomic claim via `.filter('metadata->>stock_restored', 'neq', 'true')` cegah double-restore.
- **[FIX] `src/routes/payments.ts` (partial points):** Simpan discount di metadata only — `total_amount` tidak diubah permanen.
- **[FIX] `src/routes/digital.ts`:** Tambah auth guard `/api/digital/deposit` (admin-only) + `/api/digital/order` (auth required). Hapus API key prefix dari log.
- **[FIX] `src/routes/transactions.ts`:** Cancel endpoint — ownership check jika auth header ada (guest path unchanged).
- **[FIX] `src/store/useCartStore.ts`:** Quantity clamp `Math.max(0, Math.floor(Number(quantity) || 0))`.
- **[FIX] `server.ts`, withdrawals.ts, stock.ts, productReturns.ts:** Rename `getUserId` → `getToken`.
- **[FIX] `src/routes/withdrawals.ts`:** Guard status terlalu strict (`!== "pending"`) — block flow `approved→paid`. Ganti jadi cek terminal state (`paid`/`rejected`).
- **[FIX] `src/services/background-jobs.js`:** Hapus `.is("receipt_image", null)` dari auto-cleanup — transaksi dengan receipt AI-tolak tidak pernah di-cleanup. Ganti: hanya skip jika receipt_image ada & belum pernah gagal verifikasi.
- **[FIX] `src/services/stock.js` (restoreTransactionStock):** Guard `!tx?.metadata?.stock_deducted` terlalu ketat — metadata mungkin gagal di-set (`undefined`). Ganti `=== false` agar undefined tetap proceed ke fallback `transaction_items`.
- **[NEW] `fix_lost_stock.sql`:** Recovery script untuk mengembalikan stock yang terlanjur hilang akibat bug auto-cleanup. Jalankan di Supabase SQL Editor.

v5.2.1 — Stock Accuracy Fix — RCAs & Optimistic Locking (PATCH)
----------------------------------------------------------------------------------
### Root Cause Analysis (RCA):
- **[RCA] 6 stock mutation paths ditemukan:** Sale deduction (transactions.ts), stock restore (stock.js), restock approval (stock.ts), return approval (productReturns.ts), stock opname (AdminStockOpname.tsx), iPaymu callback (payments.ts).
- **[RCA] Race condition pada SELECT→UPDATE:** Semua path menggunakan pola non-atomik `SELECT stock → compute → UPDATE stock` → overselling & phantom stock.
- **[RCA] Partial deduction lost:** Jika sebagian item gagal deduct, `stock_deducted = false` → `restoreTransactionStock` skip SELURUH items → stock yang sudah deduct tidak pernah dikembalikan.
- **[RCA] Double restore race:** Guard `stock_restored` dibaca lalu di-set secara non-atomik → dua caller bisa restore bersamaan → phantom stock.
- **[RCA] iPaymu + auto-cleanup race:** Auto-cleanup restore stock lalu iPaymu callback set status "paid" tanpa re-deduct → phantom stock permanen.

### Perbaikan:
- **[FIX] `src/services/stock.js`:** Rewrite `restoreTransactionStock` — pakai `deducted_products` dari metadata (per-item) + optimistic locking `(stock = oldStock)` pada UPDATE. Tambah `deductTransactionStock()` untuk re-deduct setelah auto-cleanup.
- **[FIX] `src/routes/transactions.ts`:** Deduksi stok sekarang atomic — UPDATE dengan `WHERE stock = oldStock`. Simpan `deducted_products` map per-item (produk ID → quantity + seller_id) di metadata. Partial deduction tetap tercatat untuk restore.
- **[FIX] `src/routes/payments.ts`:** iPaymu SUCCESS callback — jika `stock_restored = true` (auto-cleanup sudah restore), panggil `deductTransactionStock()` untuk re-deduct. iPaymu FAILURE callback — sederhanakan, hapus guard redundant (handle di `restoreTransactionStock`).
- **[FIX] `src/routes/stock.ts`:** Restock approval — optimistic locking `(stock = oldStock)`.
- **[FIX] `src/routes/productReturns.ts`:** Return approval — optimistic locking + hapus `Math.max(0, ...)` yang silent data loss.
- **[FIX] `src/pages/dashboard/admin/AdminStockOpname.tsx`:** Stock opname — optimistic locking, tolak jika stock sudah berubah sejak dimuat.
- **[FIX] `src/routes/admin.ts`:** Stock report — tetap pakai dual-query untuk initialStock akurat.
